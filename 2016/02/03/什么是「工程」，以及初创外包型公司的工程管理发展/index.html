从「匹配」说起

手推磨产生的是封建主的社会，蒸汽磨产生的是工业资本家的社会。
火药必然摧毁中世纪骑士阶层，钟表的产生必然会把效率观念带入人类思想。

上面要说的就是「匹配」的问题，大意就是什么样的技术结构，就会带来什么样的社会结构，这有固然些技术决定论的论调，但是历史上的确有不少如上所述的现象。
反过来也有类似的变革，如果你主动寻求变革，率先改变了社会结构，那么它反过来也会对你的技术结构提出要求，如果你的技术结构满足不了，那它就支撑不了你的社会结构变革。

厘米脚印的变化

从软件作坊式的生产，到工厂式的生产。

在进入主线剧情之前，从规模上说，厘米脚印整体算是一个手工作坊式的咨询公司。这个时有一个自己简单的静态官网，客户，freelancer等都是线下沟通，有一个比较简单的作品列表，而且入口不在官网门面上，接的项目也多是生命周期不是很长的项目。

我虽然只经历了一段很短的这个时期，但是认为这个时候公司工程管理策略，同样是与公司规模像匹配的手工作坊式管理，这种线下业务的形式比较灵活，也就是说，没有一套很严格的规范来约束。出了问题就补救一下，这样一直也没什么大事故发生，基本能满足需求。

但是后来就慢慢不同了，除工程管理外的其他业务开始规范，PM们谈项目的整个流程规范都慢慢出现和开始被遵守了。但是这些规范对工程本身还没有直接的约束，体现的不那么明显。但是从官网开始上线之后，这个约束就慢慢强起来了，官网是门面，它天然就有一种很强的约束功能。

也就是从这个时候，运维的工作量迅速上升。之前的工程管理缺陷立马就暴露出来了，最明显的几点就是：找不到代码在哪，代码中的依赖竟然没有版本号，作品部署的哪里都是，有的在开发服务器上，有的在freelancer服务器上，有的在产品服务器上，然而什么样的作品放到什么服务器上，又怎么区分，这些我们也都没有概念。

而且自从官网上放上作品列表之后，服务的容灾也变得非常重要起来，除了问题不能想以前那样缝缝补补了，有些问题甚至立刻都解决不了，这不是态度上或是工作既不积极的问题，而是这种技术风险总是存在的，没有容灾，出现比较棘手的问题时只能狗带。

什么是「工程」？

什么是工程？从字面意思来看，「工程」中的「工」字很好理解，但「程」字的意思就不那么明显。可以看另外一些词——「章程」，「时程」，「里程」…等等，在这些词汇中，「程」字的含义基本就是「规则」，「某种延续性的积累」。而工程就可以从字面意思上理解为：某种带有延续性，规则性的工作。

当然上面只是字面意思，事实上，什么是「工程」就像什么是「科学」那样难以定义和解释。这是因为它们并不像我们直觉上认为的那样，不仅仅是一种「纯粹客观」的活动，而是受我们已有经验和知识结构的影响，限制的。这些概念难以清晰的定义，那么我们如何更好的去理解它们？一个办法就是将目前这些概念并不是很清晰的事物与其他事物进行对比，在这种区分中去更好的认识它，其实「概念」本身也是这么来的。

爱因斯坦说过一句他对科学与工程的理解：

“Scientists investigate that which already is;
Engineers create that which has never been.”

科学家探究已经存在的（事物）;工程师创造从未有过的（事物）。

那么什么是工程？工程是一种延续性的，有秩序的创造性行为，这些行为并不是随性而为，它综合了数学，物理学，经济学等知识，并且最终的目的是创造出对现实有所影响的事物。就像盖一间房子，

厘米脚印工程管理的现状

厘米脚印的现状就是：已经趋于规范的项目流程，以及仍然比较混论的工程管理。

工程管理的两个维度：

我们平时对工程师有所要求，给出了代码风格，接口规范等要求。如果这样的要求能被彻底贯彻执行，那么我们可以编写出不错的应用，它模块划分清晰，扩展性良好。

但是即使工程师们写好了一个不错的应用，并不是万事大吉了，厘米脚印涉及的项目众多。我们不止是要写好每一个项目，同时还要很好的把它们管理起来。

用一个例子来说，我们不仅要建一座好房子，还要把这些房子统一管理起来，组织成一个井然有序的城镇。

我们的确在强调如何建好一栋房子，把它建的井井有条，经久耐用，在这所房子内，你可以直接区分哪里是厕所，哪里是厨房，哪是仓库，甚至你可以根据你的喜好来非常容易的扩展这个房子。

缺席的「管理」，厘米脚印目前是有工程，无管理。
